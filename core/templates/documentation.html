HistoryHealer Documentation

This page explains the detailed process of how HistoryHealer works, including all API endpoints and data flow.

Scan Process

1. Fetching Library Media Info

API Endpoint: get_library_media_info

For each library section ID in your configuration, HistoryHealer fetches all media items to build a reference database of what's in your Plex libraries.

This data is stored in memory during the scan process and used for comparison with history items.

2. Fetching Watch History

API Endpoint: get_history

Parameters:

  • Date range (from your configuration)
  • Media types: movies and episodes
  • Maximum 100,000 items

This retrieves your complete watch history within the specified date range.

3. Processing History Items

Each history item is processed to determine if it can be matched to a library item:

  • For Movies: Attempts to match the rating_key with library media
  • For Episodes: Attempts to match the grandparent_rating_key (show's rating key) with library media

Unmatched items are stored in the SQLite database for later fixing.

Data Storage

All unmatched items are stored in the SQLite database with their complete JSON data, including:

  • Rating keys
  • Titles
  • GUIDs
  • Media types
  • Additional metadata

Fix All Process

Process Overview

When you click "Fix All Unmatched", the following process occurs for each unmatched item:

For Movies:

  1. Search Step:
    • API Endpoint: search
    • Search Query: Movie title only (without year)
    • Fallback Search: If no match found and title contains a colon (e.g., "Glass Onion: A Knives Out Mystery"), tries searching with just the part before the colon (e.g., "Glass Onion")
    • Filtering: Filters results based on the configured Section IDs in Settings. Only items from these Plex library sections will be considered for automatic fixing.
  2. Matching Step:
    • Tries to match by the movie's GUID. An exact GUID match is required for the fix to proceed. If no exact GUID match is found, the automatic fix for this movie will fail.
  3. Update Step:
    • API Endpoint: update_metadata_details
    • Parameters: old rating key, new rating key, media type

For TV Episodes:

  1. Search Step:
    • API Endpoint: search
    • Search Query: Show title only (grandparent_title)
  2. Matching Step (Two-Attempt Process):
    1. Attempt 1: Match Episode by Episode GUID
      • Searches for the episode using its own title and its specific episode GUID. Simplified title variations (e.g., removing content after a colon) may be attempted if the initial search fails.
      • If an exact match for the episode's GUID is found, this match is used.
    2. Attempt 2: Match Show by Show GUID (if Episode Match Fails)
      • If no direct episode GUID match is found, the system then searches for the TV show using the show's title (grandparent title) and the show's specific GUID. Simplified title variations (e.g., removing content in brackets) may be attempted if the initial search fails.
      • If an exact match for the show's GUID is found, this show match is used to update the episode.
    3. If neither attempt results in an exact GUID match (either for the episode itself or its parent show), the automatic fix for this episode will fail.
  3. Update Step:
    • API Endpoint: update_metadata_details
    • Parameters: old rating key, new rating key, media type

Individual Fix Button

The individual "Fix" button performs the exact same process as "Fix All" but for a single item:

  1. Retrieves the unmatched item from the database
  2. For Movies:
    • Searches for the movie title (without year)
    • If no match found and title contains a colon, tries searching with just the part before the colon
    • Filters results based on the configured Section IDs in Settings during automatic fix attempts.
    • Tries to match by GUID, then by year
    • Updates metadata if a match is found
  3. For Episodes:
    • Searches for the show title
    • Tries to match by GUID
    • Updates metadata if a match is found

Manual Fix Process

The "Manual Fix" button provides a step-by-step interface for fixing items with your guidance:

Step 1: Search

  • API Endpoint: Tautulli's search command is used internally.
  • For Movies:
    • The search is performed for movie titles.
    • The default search query is the original movie title from the history item. You can modify this query.
    • Search results will include items from all libraries. The automatic filtering by Section IDs (configured in Settings) is not applied during manual search, allowing you to see all potential matches.
    • The system indicates which results match by GUID or year to help you choose the correct item.
    • The selected match will provide its rating_key for the fix.
  • For Episodes:
    • You are given the option to search for either show titles or episode titles.
    • When searching by Show Title:
      • The default search query is the original show title (grandparent title) from the history item. You can modify this query.
      • The selected match (a TV show) will provide its rating_key. This key is used as the grandparent_rating_key for the episode being fixed, effectively linking the episode to the chosen show.
    • When searching by Episode Title:
      • The default search query is the original episode title from the history item. You can modify this query.
      • The selected match (an episode) will provide its specific rating_key. This key is used directly to fix the episode.
    • For both episode search types, results include items from all libraries, bypassing the automatic Section ID filtering.
  • You can always modify the suggested search query and view the raw API response from Tautulli for troubleshooting or advanced matching.

Step 2: Update Metadata

  • API Endpoint: /api/manual-fix/update (HistoryHealer internal API)
  • This process updates HistoryHealer's local database, re-linking the unmatched history entry to the new_rating_key of the Plex item you selected. It also attempts to call Tautulli's update_metadata_details command to ensure the fix is reflected in Tautulli itself. The success or failure of the Tautulli update will be noted in the fix details.
  • When you select a match, the application sends to its backend:
    • Old rating key (from history)
    • New rating key (from search results)
    • Media type
  • You can see the raw API response

Step 3: Completion

Confirms the successful update and provides options to:

  • Return to unmatched items list
  • Start a new manual fix

Item Grouping

HistoryHealer automatically groups duplicate items to make management easier:

Grouping Logic

  • For Movies: Groups items with identical titles and years
  • For Episodes: Groups items with identical show titles (grandparent_title)
  • Only items with 2 or more duplicates are shown in groups
  • Unique items (with no duplicates) are displayed individually

Group Features

  • Expandable/collapsible groups to view individual items
  • "Fix All" button to fix all items in a group at once
  • "Ignore All" button to ignore all items in a group at once
  • Group count indicator showing how many items are in each group

Duplicate Prevention

When running a new scan, HistoryHealer prevents re-adding items that already exist in the database:

Process

  1. During a scan, each unmatched item is checked against the database
  2. Items are matched based on rating_key, media_type, and GUID
  3. For episodes, the grandparent_rating_key is also checked
  4. If an item already exists (even if it's marked as ignored or fixed), it's skipped
  5. A count of skipped items is displayed in the scan results

This prevents previously ignored items from reappearing as unmatched items in subsequent scans.
